# STeP Graph Construction Rules

A STeP kernel is a Python function `build_graph()` that returns `(graph, output_op)` where:
- `graph` is a `networkx.MultiDiGraph`
- `output_op` is the final `OffChipStore` operation

**You only need to output `build_graph()`.** Imports, constants (M, K, N, SEED, etc.), and `compute_gold()` are provided externally. Your `build_graph()` can reference these module-level constants directly.

## Required Imports
```python
import torch
from networkx import MultiDiGraph
from step_py.ops import (
    OffChipLoad, OffChipStore, BinaryMap, UnaryMap, BinaryMapAccum,
    Accum, Flatten, Reshape, Promote, ExpandRef,
    Bufferize, Streamify, DynStreamify, RetileStreamify,
    FlatPartition, FlatReassemble, Broadcast, RepeatStatic,
    EagerMerge, Parallelize,
)
from step_py.utility_ops import SelectGen, FilterLastTile, ConsumerContext
from step_py.functions import map_fn, map_accum_fn, accum_fn, init_fn
from step_py.datatype import Tile, DynTile, Stream, Float16, Float32, Uint32, Uint64, Bool, MultiHot, Index, Select
from rewrite.broadcast import infer_broadcast
```

## Operator Quick Reference
- `OffChipLoad` / `OffChipStore` — HBM ↔ on-chip tile transfer
- `BinaryMap(fn)` — element-wise binary op (Matmul, Add, Mul, Div)
- `UnaryMap(fn)` — element-wise unary op (Silu, Exp, RowWiseSum)
- `BinaryMapAccum(fn, init_fn, rank)` — fused map-accumulate (GEMM). Removes last `rank` dims.
- `Accum(fn, init_fn, accum_rank)` — reduction. Removes last `accum_rank` dims.
- `Bufferize(rank)` / `Streamify(repeat_factor, rank)` — on-chip data reuse. See operator reference for shape rules.
- `Broadcast(num_consumers)` — replicate stream. Access: `(broadcast, 0)`.
- `Promote(promote_rank)` — insert size-1 dimension.
- `ExpandRef(ref, expand_rank)` — expand size-1 dims to match reference.
- `RepeatStatic(repeat_factor)` — append new innermost dimension.
- `Flatten(min_rank, max_rank)` / `Reshape(chunk_size, reshape_rank)` — shape transforms.
- `Parallelize` / `EagerMerge` / `FlatPartition` / `FlatReassemble` — multi-consumer/MoE ops.

## Rules
1. Every graph must have at least one OffChipLoad and one OffChipStore.
2. Use `write_back_mu=False` for intermediate results to avoid unnecessary HBM round-trips.
3. Tile dimensions must evenly divide tensor dimensions (or pad).
4. `build_graph()` must return `(graph, final_store_op)`.
5. Do not use float16 unless the baseline uses it.
6. Always call `graph = infer_broadcast(graph)` before returning.
7. Both inputs to BinaryMapAccum must have identical stream shapes.
8. **Compute bandwidth budget:** The chip has a total compute bandwidth of **{total_compute_bw}**. The sum of all `compute_bw` values across all compute operators (BinaryMap, UnaryMap, BinaryMapAccum, Accum) must not exceed {total_compute_bw}. Ideally the full budget is used (sum == {total_compute_bw}). Distribute bandwidth based on which operators are on the critical path.
9. **HBM configuration:** {hbm_channel_num} channels, per-channel latency {hbm_per_channel_latency} cycles, init interval {hbm_per_channel_init_interval} cycles, outstanding {hbm_per_channel_outstanding}, start-up {hbm_per_channel_start_up_time} cycles, address offset {hbm_addr_offset} bytes. Each load/store has its own `par_dispatch`; all concurrent operators share the {hbm_channel_num} channels. Exceeding {hbm_channel_num} total is not an error but excess requests queue and add latency. Tune `par_dispatch` per-operator to prioritize critical-path data movement.
10. **On-chip scratchpad:** {on_chip_memory_human} ({on_chip_memory_bytes} bytes). Peak on-chip buffer usage (`on_chip_bytes`) must not exceed this. Larger tiles, higher `par_dispatch`, and Bufferize/Streamify all increase usage.
